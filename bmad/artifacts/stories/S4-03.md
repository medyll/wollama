S4-03 — tool_calls DB + agent status API

Goal
Persist every agent invocation in the `tool_calls` table and expose a status endpoint so the frontend can poll or subscribe to agent execution state.

Tasks
- Ensure `tool_calls` DB table is initialized in server/db/database.ts (may already exist — verify)
- Write AgentRunner service (server/services/agent-runner.service.ts):
  - Accepts { slug, input, chat_id? }
  - Creates tool_call record with status: 'running'
  - Executes agent, updates record to 'done' or 'error' with output/error
  - Returns tool_call_id immediately (async execution)
- Add GET /api/agents/:tool_call_id/status route — returns { status, output?, error? }
- Wire `/search` and `/fetch` slash commands (in skills pipeline) to route to AgentRunner
- Unit tests: AgentRunner happy path, error path, status polling

Acceptance
- tool_call record created on invocation, updated on completion
- GET status returns correct state at each lifecycle stage
- Tests cover: running → done, running → error, unknown id → 404
